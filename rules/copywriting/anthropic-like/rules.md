# 文案规则：anthropic-like

这套规则不是把中文内容改成英文企业博客，也不是模仿 Anthropic 的句子。它提取的是 Anthropic Blog 里最有价值的写作机制：真实场景、明确变化、失败模式、工作流、角色边界、证据和下一步。

对中文内容，尤其是小红书、公众号、朋友圈长文、产品说明和方法论文章，正确目标是：

> 中文内容的可读性 + Anthropic 式可信结构。

中文内容负责让人愿意读：冲突感、第一人称、短句节奏、清晰标题、可转发判断。

Anthropic-like 负责让人相信：场景、机制、边界、证据、适用范围和可执行下一步。

如果两者冲突，先保留用户原内容所属平台的可读性，再用 Anthropic-like 补足可信度。不要为了“克制”把文章写平。

还有一个优先级：如果原文里已经有好标题、好开头、好强句，不要为了显得“改写过”而重写它。先保留，再修整。真正要改的是空话、重复、跳步、缺证据的地方。

## 一句话原则

不要用口号证明重要性。用真实工作流里的变化、摩擦和证据，让读者自己感到这件事重要。

好文案要让读者看完后能复述一个判断：

> 旧做法为什么失效，新机制为什么成立，下一次该怎么做。

## 写作前先判断内容类型

不同内容有不同表面形态。不要把所有内容都写成同一种企业博客腔。

### 社媒经验文案 / 小红书 / 朋友圈长文

保留：

- 标题里的误区、反转、踩坑、真相感。
- 第一人称进入方式。
- 一个具体的冲突或失败现场。
- 4-7 个清晰小标题。经验文案必须有小标题，不能只用段落转折代替。
- 结尾的边界判断，而不是抽象升华。
- 原文里已经成立的短句和判断句。

Anthropic-like 的作用是把它从“情绪分享”升级成“可信方法”：

- 把感叹换成失败机制。
- 把口号换成具体对象。
- 把“我悟了”换成“为什么会这样、机制怎么设计、怎样证明有效”。
- 把工具或产品卖点放在机制之后，作为落地方式，不作为最后高潮；如果工具段会让主线变散，可以压缩到一小段，甚至删除。

### 方法论 / 教程 / 技术文章

保留清晰结构和可操作步骤。每个抽象概念后面都要跟真实对象，例如角色、文件、命令、权限、流程、日志、截图、指标、案例或失败成本。

### 产品文案 / 功能介绍

不要写成功能清单。写成工作流改善：

- 谁原来卡在哪里。
- 旧流程多了哪一步或风险。
- 新能力改变了哪一步。
- 使用时需要什么权限、输入或边界。
- 怎样判断它真的有用。

## 核心风格

### 1. 开头先给变化或冲突

不要从宏大背景开头。

差：

> 在 AI 快速发展的今天，协作效率已经成为每个团队必须面对的问题。

好：

> 很多人第一次尝试新工具时，都会先用最直觉的办法：把更多人、更多流程、更多自动化一起堆上去。问题通常不是出在“工具不够多”，而是出在大家没有共享同一个事实来源。

开头可以有冲突，但冲突后要马上落到真实场景。

社媒经验文案的开头不要压缩太快。优先保留这种递进节奏：

```text
场景来了。
很多人的第一反应是……
我一开始也这么想 / 也这么做。
结果不是 A，而是 B。
具体失败现场。
后来我意识到：真正问题不是 X，而是 Y。
```

不要把这些压成一句“我翻车后明白了……”。节奏太快，会少掉代入感。

### 2. 抽象概念必须落到操作边界

不要只解释“协作”“信任”“效率”“系统”“智能体”“工作流”“内容质感”这些词。解释之后马上回答：

- 谁负责？
- 信息在哪里？
- 谁能修改？
- 怎么验证？
- 怎么撤销？
- 出错时谁接手？
- 什么情况下不适用？

抽象概念只有落到边界，才不像 AI 生成的空话。

### 3. 功能不要写成功能清单

差：

> 这个方法支持协作、记录、复盘、追踪和验收。

好：

> 原来所有判断散在聊天记录里，换一个人接手就要重新问一遍。现在把状态、结论、责任人和验收证据放到同一个可追踪位置，下一位接手者不需要猜前面发生了什么。

功能清单像宣传页。工作流改善像真实经验。

### 4. 语气克制，但要有判断

这套风格不是冷冰冰的中立，也不是自嗨式营销。它会明确给建议，但建议后面要跟原因、例子或边界。

可以写：

> 一个有用的规则是：讨论可以发生在聊天里，但最终事实必须落到一个可追踪的位置。

不要写：

> 因此我们必须全面拥抱结构化协作，释放效率革命的无限潜力。

判断要直接，理由要具体。

### 5. 案例要抗 AI 味

一个好案例至少包含下面三类信息：

- 具体角色：使用者、负责人、审核者、协作者、客户、读者。
- 具体对象：文档、表格、命令、日志、截图、链接、流程、指标、样稿。
- 具体失败：覆盖、误解、遗漏、延迟、权限混乱、验收口径不一致。
- 具体修复：谁负责、放在哪里、如何检查、如何回滚。
- 具体证据：测试通过、命令输出、截图、发布链接、变更清单、数据变化。

如果案例只有“更高效、更丝滑、更智能、更有质感”，就还不够。

## 推荐结构

### 社媒经验文案

适用于踩坑、复盘、经验总结、观点文、小红书笔记、朋友圈长文。

这是格式契约，不是建议。除非用户明确要求极短帖，否则输出必须包含：

- 一个标题。如果原标题已经有误区、反转、踩坑或真相感，优先保留原标题，只删表情或冗余词。不要改成长副标题。
- 一个冲突开头。
- 一个核心判断。
- 4-7 个 Markdown 二级标题，格式必须是 `## 1. 小标题`、`## 2. 小标题`。
- 一个适用边界或收束段。

不允许只用加粗句代替小标题。`**小标题**` 不算合格小标题。

```markdown
# 标题：误区 / 反转 / 具体对象 / 结果

开头：
具体场景来了。
很多人会怎么想。
我或某类人也曾这么想 / 这么做。
结果不是 A，而是 B。
具体失败现场。

核心判断：
真正的问题不是 A，而是 B。

## 1. 用判断句命名第一个失败机制

解释为什么旧做法会失败。必须有具体场景或具体对象。

## 2. 用判断句命名第一条新边界

解释新机制如何约束问题。必须让读者知道下一次怎么做。

## 3. 用判断句命名第二条新边界

继续展开角色、责任、范围、流程或信息位置。

## 4. 用判断句命名验证方式

说明怎样证明这件事真的完成、真的有效或真的可交接。

## 5. 用判断句命名工具 / 流程如何落地产物

如果有工具、产品或流程亮点，放在这里。它只能服务前面的机制，不能变成广告。

适用边界：
什么情况下不需要这套方法。
什么情况下必须先搭机制。

收尾：
短句判断，回到核心机制。
```

要求：

- 不要把社媒文案压成碎片清单。
- 不要把标题改成冷静论文题。
- 不要把一个好标题改成长标题。标题越长，传播感通常越弱。
- 不要删掉第一人称或真实冲突。
- 小标题要服务判断，不要写成“优势一、优势二、优势三”。
- 小标题必须是机制判断句，不是栏目名。
  - 差：`## 1. 背景`
  - 差：`## 2. 方法`
  - 好：`## 1. 问题通常不是工具不够，而是事实来源太散`
  - 好：`## 2. 一个机制如果不能验证，就还只是口头约定`
- 如果生成结果没有 4-7 个 `##` 小标题，直接判定为不合格，必须重写。
- 小标题要短一点，像给机制命名。优先使用 8-18 个汉字的判断句。
  - 差：`## 1. 并行不会自动变快，事实来源太散只会更乱`
  - 好：`## 1. 事实来源不能散`
  - 好：`## 2. 文件要承担秩序`
  - 好：`## 3. 主控方不要下场乱写`
- 小标题不要把全文观点提前讲完。标题负责命名机制，正文负责解释机制。

### 方法论 / 教程

适用于系统讲一个方法、流程、工具实践。

```text
问题：旧方式在哪里开始失效。
失败模式：列出具体摩擦。
机制：解释新方法如何工作。
操作：给出能照做的步骤。
验证：说明怎样证明有效。
边界：说明什么时候不该用。
下一步：给检查清单或开始动作。
```

### 产品说明

适用于介绍工具、功能、产品能力。

```text
谁遇到什么问题。
旧流程为什么麻烦。
新能力改变哪一步。
用户需要提供什么输入或权限。
给一个真实使用场景。
说明限制和适用范围。
给下一步动作。
```

## 句子规则

- 多用“原来 / 现在 / 所以”的因果链，少用“颠覆 / 重塑 / 赋能”。
- 多写“失败会出现在……”而不是“这件事很重要”。
- 多用可检查对象：文件、命令、权限、日志、截图、角色、流程、指标、样稿、链接。
- 句子可以短，但不要碎成口号。
- 第一人称可以用，但要来自真实经历或明确角色，不要伪装成体验。
- 每个小节只服务一个判断。
- 强句要短。例如：“不是更快，而是更乱。”“没有证据，就还没完成。”“先定边界，再谈协作。”
- 社媒经验文案的小标题也要是强句。不要写成中性栏目名。
- 工具亮点要放在机制之后。先讲为什么需要它，再讲它改变了哪一步。
- 如果出现“最爽的是”“大招”“一键搞定”这类表达，先判断它是否破坏主线。能删就删；必须保留时，再补工作流上下文：原来哪里不可协作，现在如何可编辑、可追踪、可复用或可验证。
- 不要主动发明新场景来显得完整。优先使用原文已有细节；只有原文太空时，才补最小必要例子。

## 常用替换

- “这很重要” -> “问题通常会在某个具体环节暴露。”
- “提升协作效率” -> “少了一次反复确认状态的人工对齐。”
- “形成统一认知” -> “所有人只看同一个事实来源。”
- “最佳实践是” -> “一个有用的规则是”。
- “未来趋势” -> “采用前先问这几个问题。”
- “能力很强” -> “它具体少掉了哪一步。”

## 禁用倾向

- 不写“未来已来”。
- 不写“范式转移”“极致体验”“无限可能”。
- 不堆三个抽象收益。
- 不用“这不仅是 X，更是 Y”作为主论证。
- 不把文章写成“我刚才做了什么”的过程旁白，除非用户明确要求复盘。
- 不为了显得高级，删掉中文内容里的冲突感、第一人称入口和可读节奏。
- 不把“anthropic-like”理解成“更冷淡、更少标题、更像新闻稿”。
- 不用加粗段落代替 Markdown 小标题。社媒经验文案默认要有 `## 1.` 这种编号标题。
- 不把工具段写成宣传口播。工具只负责让前文机制落地。
- 不让工具段成为最后的高潮。最终高潮应该是方法判断，而不是产品能力。
- 不为了保留原文所有信息而牺牲主线。如果某个工具、产品或卖点让文章变散，可以弱化或删除。
- 不过度润色。满意的社媒经验文案经常是朴素、有判断、有事实，不是每段都很“会写”。
- 不把信息密度高的内容压缩成“经验摘录”。

## 自检标准

生成后逐条检查：

1. 读者能否用一句话复述核心判断？
2. 开头有没有真实变化、冲突或失败现场，而不是抽象背景？
3. 每个重要概念后面是否跟了具体对象？
4. 是否保留了目标平台的可读性，而不是统一改成企业博客腔？
5. 工具或产品亮点是否服务于主机制，而不是突然变成广告？
6. 结尾是否留下边界、清单、问题或下一步，而不是空泛升华？
7. 输出是否能直接发布，而不是像一份规则总结？
8. 如果是社媒经验文案，是否有 4-7 个 `##` 编号小标题？如果没有，必须重写。
9. 小标题是否是机制判断句，而不是“背景 / 方法 / 总结 / 优势”这类栏目名？
10. 是否保留了原文里最有传播感的标题、开头和强句？如果都被重写平了，必须回退。
11. 工具或产品段是否抢走主线？如果抢走了，压缩或删除。

## 结尾模板

好的结尾通常像这样：

- “先问一个问题：下一个人接手时，能不能继续做？”
- “如果一个结果不能复现，它就还不算完成。”
- “不要先堆工具。先确定它解决的是哪个真实摩擦。”
- “小问题，用简单方法。复杂问题，先搭机制。先把问题分清，再决定方法。”
