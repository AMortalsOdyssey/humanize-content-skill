# 文案规则：anthropic-like

这套规则来自对 Anthropic 官方 Blog 多篇文章的风格归纳。不要模仿英文句子，也不要把文章写成“企业新闻稿”。要提取它的写作机制：先讲真实变化，再讲失败模式、工作机制、角色边界、可验证证据和下一步问题。

## 一句话原则

不要用口号证明重要性。用真实工作流里的变化、摩擦和证据，让读者自己感到这件事重要。

如果输入本身是一段经验分享或方法论文案，优先写成“可转发的短文章”，不要为了像社媒而降维成碎片清单。好文案的目标不是更短，而是让读者看完后能复述一个判断：旧做法为什么失败，新机制为什么成立，下一次该怎么做。

## 核心风格

### 1. 开头先给变化，不先喊口号

常见开头不是“AI 正在重塑世界”，而是：

- 以前人们怎么做。
- 现在什么东西变了。
- 这个变化带来了哪个具体问题。

例如：

> 过去，和 AI 协作通常是一个人面对一个聊天窗口。真正麻烦的地方出现在多人和多个 Agent 需要共享同一个目标的时候。

这种开头比“多 Agent 是未来趋势”更可信，因为它先给读者一个能理解的工作变化。

### 2. 概念要快速落到操作边界

不要只解释“Agent 身份”“记忆”“协作空间”“事实来源”这些概念。解释之后马上落到：

- 谁拥有权限。
- 信息放在哪里。
- 哪些文件或频道是事实来源。
- 谁能修改。
- 怎么撤销。
- 怎么审计。
- 怎么验证。

抽象概念只有落到边界，才不像 AI 生成的空话。

### 3. 产品或方法不要写成功能清单

不要写：

> 这个方法支持多人协作、任务拆解、状态同步和结果验收。

改成：

> 原来每个 Agent 都在自己的上下文里推进，最后集成时才发现理解不一样。现在先把需求矩阵、接口契约和验收证据放进同一个协同区，Agent 做完后不是说“完成了”，而是留下能复跑的命令或截图。

功能清单像宣传页。工作流改善像真实经验。

### 4. 语气克制，但要有判断

这套风格不是冷冰冰的中立，也不是自嗨式营销。它会明确给建议，但建议后面要跟原因、例子或边界。

可以写：

> 一个有用的规则是：聊天可以讨论，但不能当事实来源。

不要写：

> 因此我们必须全面拥抱结构化协作，才能释放 AI Agent 的最大潜力。

判断要直接，理由要具体。

### 5. 案例要抗 AI 味

一个好案例至少有这些元素里的三个：

- 具体角色：主 Agent、子 Agent、团队负责人、审稿人、设计 owner。
- 具体对象：需求矩阵、契约文件、日志、截图、命令、PR、会议纪要。
- 具体失败：互相覆盖、方向漂移、权限混乱、验收口径不一致。
- 具体修复：谁负责、文件放哪里、如何检查。
- 具体证据：测试跑绿、命令输出、截图、发布链接、变更清单。

如果案例只有“效率提升、协作更顺畅、体验更好”，就还不够。

## 推荐结构

### 教程 / 方法论

优先使用这条结构，尤其适用于 Agent 协作、工程方法、团队流程、工具实践这类主题：

1. 误区开头：很多人原来怎么做，为什么这个想法看起来合理。
2. 失败现场：给 2-4 个具体摩擦点，不要只说“翻车了”。
3. 核心判断：用一句话说清楚真正问题是什么。
4. 机制展开：按 4-6 个小节讲事实来源、角色边界、写入范围、验证方式、集成/回滚。
5. 工具落地：如果有工具亮点，只写它如何把机制变成真实产物，不要把工具段写成广告。工具段通常只能占全文 10%-20%，不能抢走主线。
6. 适用边界：什么时候一个人/一个 Agent 就够，什么时候才需要这套系统。
7. 强判断收尾：用短句收住，不要硬升华。结尾要回到核心判断，不要停在工具卖点上。

一个稳定骨架：

```text
误区：大家以为 X 会更快。
失败：实际出现 A/B/C。
判断：真正要解决的不是 X，而是 Y。
机制 1：事实来源。
机制 2：协作秩序。
机制 3：角色边界。
机制 4：验证证据。
工具落地：工具如何把上面机制变成团队可协作产物，只作为主机制的延伸。
收尾：小需求怎样，复杂需求怎样。不要反过来。
```

不要把“机制展开”写成普通功能清单。每一节都要有“为什么会失败 → 现在怎么约束 → 读者能照做什么”。

### 产品说明

1. 谁遇到了什么问题。
2. 旧流程卡在哪里。
3. 新功能改变了哪一步。
4. 需要哪些权限、数据或配置。
5. 给一个真实使用场景。
6. 使用前要检查什么。

### 小红书 / 社媒短文

1. 用一个真实误区或踩坑开头。
2. 给一个短失败现场。
3. 抽出背后的机制。
4. 给几条读者能照做的规则。
5. 用个人判断收尾，不要硬升华。

如果原始素材信息密度高，不要强行压短。保留文章骨架，只把句子变得更口语、更可读。小红书感来自开头钩子、段落节奏和判断密度，不来自删掉论证。

## 句子规则

- 多用“原来 / 现在 / 所以”的因果链，少用“颠覆 / 重塑 / 赋能”。
- 多用“失败会出现在……”而不是“这件事很重要”。
- 多用“文件、命令、权限、日志、截图、角色、流程、指标”这类可检查对象。
- 句子可以短一点。不要每句都写得很完整、很圆。
- 第一人称可以用，但要来自真实经历，不要伪装成“亲身体验”。
- 结尾给问题、清单或下一步，不要给口号。
- 每个小节只服务一个判断。不要把两个机制硬塞进同一节。
- 强句要短。例如：“结果不是快，而是乱。”“聊天不是事实来源。”“不要反过来。”
- 工具亮点要放在机制之后。先讲为什么需要它，再讲它改变了哪一步。
- 如果出现“最爽的是”“大招”“直接当 PPT 用”这类表达，必须补上工作流上下文：原来产物为什么不可协作，现在产物如何可编辑、可评论、可复用。
- 工具段写法：先说“前面的协作文件如果只停在本地，团队还是接不住”，再说“用某个工具把矩阵、方案、流程图、状态更新放进团队协作环境”。最后一句必须回到“事实来源、边界、证据”。

## 常用替换

- “这很重要” -> “问题通常会在集成时暴露。”
- “提升协作效率” -> “少了一次反复确认状态的人工对齐。”
- “形成统一认知” -> “所有人只看同一份需求矩阵。”
- “最佳实践是” -> “一个有用的规则是”。
- “未来趋势” -> “采用前先问这几个问题”。

## 禁用倾向

- 不写“未来已来”。
- 不写“范式转移”“极致体验”“无限可能”。
- 不堆三个抽象收益。
- 不用“这不仅是 X，更是 Y”作为主论证。
- 不把文章写成“我刚才做了什么”的过程旁白，除非用户明确要求复盘。
- 不为了加入新卖点破坏主线。新卖点必须回答同一个核心问题。
- 不把工具段写成宣传口播。工具只负责让前文机制落地。
- 不让工具段成为最后的高潮。最终高潮应该是方法判断，而不是产品能力。
- 不把满意的文章骨架压缩成“经验摘录”。如果原主题有复杂系统属性，保留编号小节。

## 自检标准

生成后逐条检查：

1. 读者能否用一句话复述核心判断？
2. 开头有没有真实失败现场，而不是抽象背景？
3. 每个小节是否都有具体对象，例如文件、命令、schema、截图、owner、画板、表格？
4. 工具亮点是否服务于主机制，而不是突然变成广告？
5. 结尾是否足够干净，能留下一个可执行边界？
6. 和输入相比，是否减少了口号和表情，增加了失败细节、角色边界和验证方式？

## 结尾模板

好的结尾通常像这样：

- “先问一个问题：下一个人打开文件后，能不能继续做？”
- “如果一个结果不能复现，它就还不算完成。”
- “不要先多开 Agent。先确定它们共享同一个事实来源。”
- “小需求，一个 Agent 加清晰验收就够了。复杂需求，先搭协同区，再开 Agent。”
